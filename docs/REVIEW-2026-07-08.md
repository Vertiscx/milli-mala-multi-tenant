Three independent review passes (security/tenant isolation, core-flow

## Deferred — decided, tracked

### H5 — Duplicate-apiKey guard is dead code in production (deferred 2026-07-08)

The cross-tenant `malaskra.apiKey` uniqueness check (SYN-MUT-28-1) lives only
in `FileTenantStore.fromJson()` (`src/tenant.ts`). The Node entrypoint
constructs `new FileTenantStore(loadTenants())` directly and the Worker uses
`KvTenantStore` — neither path runs the check. If two tenants are provisioned
with the same `*_MALASKRA_API_KEY`, a holder of tenant A's key can
authenticate `/v1/attachments` and `/v1/cases` requests as tenant B
(contained only by the ticket-brand cross-check).

**Operational rule until fixed: `MALASKRA_API_KEY` values MUST be unique per
tenant.** Tracked as a planning todo (`enforce-duplicate-apikey-guard-in-prod`);
the fix interacts with the pending per-tenant graceful-loading work.

### H3 — Multi-file upload is not idempotent; retries create duplicates (parked 2026-07-08)

`uploadDocument` uploads the ticket PDF, then each attachment per-request,
throwing on the first failure. The webhook then returns 500, Zendesk
auto-retries, and the retry re-uploads the PDF and already-succeeded
attachments — duplicate documents in the OneSystems/GoPro case. A plain
double-fire of the webhook duplicates the full document set (no replay guard).

Parked pending discussion with the archive side: do OneSystems/GoPro dedupe
or tolerate duplicates? If not, options are per-file error collection (no
throw-on-first), an idempotency key per document, or a gateway-side
"already documented" guard. Tracked as a planning todo
(`archive-upload-idempotency`).

## Medium — known, accepted for now

1. **No timeout on outbound fetches** (`onesystems.ts`, `gopro.ts`, `zendesk.ts`) — a hung upstream pins the request; Zendesk's webhook timeout then retries, stacking concurrent hung uploads. Fix: `AbortSignal.timeout(~30s)` helper.
2. **Attachments skipped by `maxFiles`/`maxTotalBytes` limits are not recorded** (`src/zendesk.ts`) — on the file-count limit the remaining files vanish without `.failed` entries; on the size limit only the tripping file is recorded. Note + audit under-report.
3. **`orphan_case` response leaks raw downstream error text** (`src/cases.ts`, 207 body `error` field) — inconsistent with the generic-message sanitization everywhere else.
4. **Zendesk client errors discard the response body** and use `statusText` (empty over HTTP/2) — failures surface as `Zendesk API error: 422 ` with the detail lost.
5. **No transient/permanent error distinction** — 400/401/429/503 from doc systems all become the same thrown error → webhook 500 → Zendesk retries permanent failures; no 401→re-auth retry inside the token TTL.
6. **Worker memory ceiling** — up to 100 MB of attachments buffered, then base64-encoded (×1.37) into a joined body string; Cloudflare isolates cap at 128 MB, so ~40 MB of real attachments can OOM a Worker deployment. Current production deployments are Docker/K8s; revisit before any Worker deployment handles attachment-heavy tenants.
7. **Webhook replay window** — signature freshness is ±5 minutes with no nonce; a captured signed webhook can be replayed for ~10 minutes, duplicating archival. Accepted; revisit together with H3.
8. **Full upstream response bodies logged at info, unbounded** (`src/onesystems.ts`) — deliberate debugging aid (#20), but responses can carry case metadata / kennitala-adjacent PII and error bodies are embedded unbounded in thrown errors. Consider a size cap (~2 KB) and field filtering.
9. **Trailing-slash `baseUrl` still unguarded** — `${baseUrl}/api/...` joining means a configured trailing slash 404s every upload (bit staging once already). Fix: strip in config validation.
10. **Post-back can re-fire the originating trigger** — the GW-01 ticket PUT may match tenant trigger conditions and loop (documentation → note → trigger → webhook). Tenant triggers MUST NOT fire on ticket-update events that the post-back itself produces. A failed post-back is also only warn-logged (result lost, no retry/audit flag).

## Low

- `caseNumberSource` misclassified for genuine case numbers starting `ZD-`; the `create_failed` finalize records `source: created` with a fallback number.
- `Number(body.ticket_id)` accepts booleans/whitespace (`true` → ticket 1).
- `escapeXml` applied to the `xml` metadata field would corrupt real XML (latent — no current caller passes `xml`).
- Non-JSON 200 from OneSystems is coerced to `{success: true}`; the GoPro equivalent throws a bare `SyntaxError` without context.
- Unbreakable long tokens (URLs, base64 blobs) overflow the PDF page width.

## Verified sound (positives)

- Audit + webhook auth are timing-safe with unset-secret guards; KV-prefix injection blocked via `sanitizeAuditParam`.
- SSRF domain check on attachment URLs (exact last-two-label match) is correct.
- CRLF sanitization in the multipart builder prevents boundary/header injection.
- No security-relevant divergence between the Worker and Node runtimes.
- Error precedence in `documentTicket` is deliberate and test-locked.
- UTF-8 Icelandic text is preserved end-to-end (the entity path was the only breakage, fixed as H4).
